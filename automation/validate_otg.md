# OTGマウス入力の実機確認

セットアップ（ようこそ）画面で、USB OTGマウスが動くかを確認する。  
動かない機種では「ラズパイ＝マウス」方式は使えず、物理タップ等に切り替える。

## 準備

1. 端末を工場出荷状態に戻す（または初期設定のようこそ画面まで進める）
2. USBスイッチまたはOTGアダプタで、端末に **USBマウス**（またはHID化したPico/Pi）を接続
3. 端末はUSBホスト（OTG）側になること（PCのADBには繋がない）

## 確認チェックリスト

- [ ] ようこそ画面でカーソル（またはタップ相当）が見える／反応する
- [ ] 画面上の同一点を連続クリックできる
- [ ] **同じ場所を6回クリック**するとQRスキャナー（または追加セットアップ）が開く
- [ ] スキャナーが開いたあと、固定QRが画角に入っていれば読める

## 結果の記録

`automation_config.json` の `otg` セクションに書く:

```json
"otg": {
  "validated": true,
  "device_model": "Pixel 7",
  "android_version": "14",
  "notes": "ようこそ中央タップでカーソル反応あり。6タップでQR起動OK"
}
```

`validated` が `false` のまま本番 `orchestrate.py` を走らせると警告が出る。

## うまくいかないとき

| 症状 | 対処 |
|------|------|
| 全く反応しない | その機種はセットアップ中OTG無効の可能性大 → 物理ソレノイド検討 |
| 充電だけしてデータ無し | OTG対応ケーブル／アダプタを使う |
| 6タップしてもQRが出ない | タップ間隔を空ける（0.3〜0.8秒）、座標をずらして再キャリブ |
| QRは出るが読めない | [jig_qr_setup.md](jig_qr_setup.md) の距離・明るさを見直す |

## ADB生存中の補助確認（任意）

通常起動・USBデバッグONの状態なら、OTG切替前にマウスのHID列挙だけPC側で確認できる場合がある:

```bash
# macOS: 接続デバイス一覧（参考）
system_profiler SPUSBDataType | head -80
```

**決定的な確認は必ずようこそ画面で行うこと。**
